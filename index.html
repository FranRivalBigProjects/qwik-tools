How Qwik Resumability Works

Step 1: Server Serializes Everything Into HTML

When Qwik renders on the server, it does everything React SSR does — renders HTML — but additionally serializes the complete application state and event handler references directly into the HTML as JSON in a special script tag (type="qwik/json"). Event listeners are encoded as on:click attributes pointing to lazy-loadable chunk URLs. The client receives fully-rendered HTML with all the information needed to resume — not restart — the app.

Step 2: Qwikloader Intercepts Events (No App Code Runs)

Instead of loading your app's JavaScript, Qwik loads a single 1KB inline script called Qwikloader. Qwikloader attaches a single global event listener using event delegation. It intercepts user interactions (clicks, inputs, keypresses) before any app code runs. No component code executes on startup. No virtual DOM is constructed. No event listeners are attached per-element. This is why Qwik's TTI is effectively zero regardless of app size.

Step 3: Lazy-Load Only What Users Trigger

When a user clicks a button, Qwikloader reads the on:click attribute, which contains a URL to the specific JavaScript chunk for that handler. Qwik fetches only that handler chunk (typically 2-10KB), deserializes the relevant state from the HTML JSON, and executes the handler. Adjacent components, unrelated handlers, and unvisited routes are never downloaded. This is fundamentally different from hydration — which must download everything before any interaction can be processed.

Why This Matters: Core Web Vitals Improvement

First Contentful Paint (FCP)

Same as any SSR framework — HTML arrives pre-rendered. No difference between Qwik and React SSR.

Largest Contentful Paint (LCP)

Identical to SSR approaches. Server-rendered content is available immediately.

Time to Interactive (TTI) — The Key Difference

React SSR: Full JS bundle downloads → parses → executes → page becomes interactive (3-8 seconds on slow 3G). Qwik: Qwikloader runs instantly, page is interactive immediately. First interaction downloads only needed chunks. Qwik achieves TTI in milliseconds.

Total Blocking Time (TBT)

React SSR: Large blocking time during bundle parse/execute on startup. Qwik: Zero blocking time on startup, minimal per-interaction. Massive TBT reduction on mobile.

Cumulative Layout Shift (CLS)

Same as SSR (pre-rendered HTML prevents layout shifts). Both frameworks are equal here.

Router and Data Loading with Qwik City

Qwik City extends Qwik with routing and server functions, both following resumability principles:

// ─── Qwik City (Router) — routeLoader$ ─── // Similar to Remix loaders but with resumability import { routeLoader$, routeAction$ } from ’@builder.io/qwik-city’; // Runs on server, serialized into HTML — NOT re-fetched on client export const useUserData = routeLoader$(async ({ params }) => { const user = await db.user.findUnique({ where: { id: params.id } }); return user; }); // Form action — also lazy, only fetched on submit export const useUpdateUser = routeAction$(async (data) => { await db.user.update({ where: { id: data.id }, data }); return { success: true }; });

Real-World Performance Scenarios

E-commerce Product Listing

Qwik approach: Server renders 100 products with prices and images. Qwikloader intercepts clicks. User clicks "Add to Cart" → handler chunk downloads (~4KB) → item added instantly. Filter button clicked → filter handler loads separately. No re-render of entire product list.

React SSR approach: Server renders products. Client downloads 280KB bundle, parses it (500ms on 3G), executes all product components, rebuilds state, attaches listeners. Now interactive. Same actions as Qwik but 2-3 seconds later on slow networks.

Impact: Qwik product pages load 10x faster on mobile, leading to higher conversion rates and lower bounce rates.

Dashboard with Complex Interactions

Qwik: Server renders dashboard with charts, tables, and filters. User sees everything instantly. Clicking chart → chart interaction handler loads. Table sort → sort handler loads. Only used features download code. A rarely-used export feature might never load its code.

React SSR: All feature code must download and parse before any interaction works, regardless of which features users actually use.

Impact: Qwik pays only for features users actually interact with, dramatically reducing bundle overhead.

Common Questions & Potential Pitfalls

Does resumability work with third-party scripts and plugins?

Yes, but with considerations. Third-party code (Google Analytics, chat widgets) runs outside Qwik's lazy system. Place them in head or use Partytown to isolate them from the main thread. This prevents third-party scripts from blocking Qwik's lazy loading.

How does resumability handle form submissions?

Forms work seamlessly. Server pre-renders the form HTML. On submit, Qwik's routeAction$ downloads the server handler code (~1-2KB), sends data, and processes the response. No app JS needed until submission.

Can you use Qwik with existing React libraries?

Qwik can integrate React components via the Qwik React adapter, but they'll lose resumability benefits and revert to hydration. For best performance, prefer Qwik-native components or libraries built for resumability.

Does resumability affect SEO?

No, SEO is improved. Server-rendered HTML is crawl-friendly. Qwik's fast TTI means crawlers finish faster. Better Core Web Vitals = better SEO rankings.
