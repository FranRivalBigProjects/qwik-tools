How resumability works

Step 1 — Server serializes everything into HTML
When Qwik renders on the server, it does everything React SSR does — renders HTML — but additionally serializes the complete application state and event handler references directly into the HTML as JSON in a special script tag (type="qwik/json"). Event listeners are encoded as on:click attributes pointing to lazy-loadable chunk URLs. The client receives fully-rendered HTML with all the information needed to resume — not restart — the app.

Step 2 — Qwikloader (1KB) intercepts events
Instead of loading your app's JavaScript, Qwik loads a single 1KB inline script called Qwikloader. Qwikloader attaches a single global event listener using event delegation. It intercepts user interactions (clicks, inputs, keypresses) before any app code runs. No component code executes on startup. No virtual DOM is constructed. No event listeners are attached per-element. This is why Qwik's TTI is effectively zero regardless of app size.

Step 3 — Lazy-load only what the user triggers
When a user clicks a button, Qwikloader reads the on:click attribute, which contains a URL to the specific JavaScript chunk for that handler. Qwik fetches only that handler chunk (typically 2-10KB), deserializes the relevant state from the HTML JSON, and executes the handler. Adjacent components, unrelated handlers, and unvisited routes are never downloaded. This is fundamentally different from hydration — which must download everything before any interaction can be processed.

Why this matters for web performance
The standard web performance metrics all improve with resumability: FCP (First Contentful Paint) — same as any SSR framework, HTML arrives pre-rendered. TTI (Time to Interactive) — near-zero, since Qwikloader handles events without app JS. TBT (Total Blocking Time) — zero on startup, since no large JS bundle parses and executes. CLS (Cumulative Layout Shift) — same as SSR (pre-rendered HTML). On slow 3G connections, React SSR apps can take 3-8 seconds to become interactive while the JS bundle hydrates. Qwik apps are interactive in milliseconds.

FAQ — Qwik resumability
Resumability is Qwik's alternative to hydration. Traditional SSR frameworks re-execute all component code on the client to re-attach event listeners (hydration). Qwik serializes the application state AND event handler chunk URLs into the HTML. A 1KB Qwikloader script intercepts events globally. When a user interacts, Qwik fetches only that specific handler's code chunk (~2-10KB). Result: zero JavaScript executed on startup, regardless of app size.
Three key performance improvements: (1) TTI near zero — no JS bundle to parse/execute before the app is interactive. (2) Constant startup cost — TTI doesn't grow as the app gets larger, unlike hydration. (3) Pays only for what's used — features never clicked by a user never download their code. On slow 3G, this is the difference between 3-8 seconds vs milliseconds. Lighthouse scores improve dramatically since no large JS bundle blocks the main thread.
Both render HTML on the server. The difference is what happens on the client: React SSR + hydration — downloads the full JS bundle, parses it, re-runs all component code, re-constructs the virtual DOM, re-attaches every event listener. Qwik resumability — downloads only qwikloader (1KB), reads serialized state from HTML, intercepts events globally. First interaction downloads only the clicked handler (~2-10KB). The performance gap grows as app complexity increases.
Qwik 1.x and Qwik City are stable and production-ready. Several companies use Qwik in production. Qwik 2.0 was announced with further performance improvements. Ecosystem maturity is smaller than React/Next.js but growing. Use cases where Qwik shines most: e-commerce (interaction-heavy, performance-critical), content sites with SPAs, dashboard apps. If you're building a new project where TTI and Lighthouse scores are critical, Qwik is a strong choice in 2025.
Qwik uses a special type="qwik/json" script tag injected into the HTML by the server. This tag contains a compact JSON representation of all reactive state (signals, stores) used on the page. When a handler is invoked, Qwik deserializes only the state relevant to that handler from this JSON. State updates trigger Qwik's fine-grained reactivity — only the specific DOM nodes that depend on the changed signal are updated, without re-rendering the entire component tree.
